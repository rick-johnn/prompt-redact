Architecture

Note: the service shape changed on 2026-05-30 from a transparent OpenAI-compatible proxy to a standalone redaction microservice. See ADR 0002 for the rationale.

Problem

Applications in regulated industries — healthcare chat, financial assistants, legal copilots, pharma research tools, GDPR-scoped products — routinely handle text that contains identifiers which must not spread to systems not cleared to hold them. A third-party LLM is the obvious case, but it is rarely the only one — and often not the most important one.

Many larger organizations already hold a legal agreement (in healthcare, a Business Associate Agreement) that permits sending protected data to one specific, approved LLM. For them the approved model call may already be covered — yet the same text keeps flowing into systems that are not: LLM observability and prompt-logging tools (frequently third-party SaaS), analytics pipelines and data warehouses, evaluation and QA harnesses where engineers inspect real traffic, and any cheaper or specialized model that is not on the approved list. Each is a place identifiers accumulate where they should not.

The leverage point is therefore a single redaction step, callable on its own: clean the text once, then use it safely across every downstream consumer — fast enough that the calling app can invoke it on the hot path of a user interaction.

Goals

  • Block raw identifiers from any downstream system the caller chooses. The service's only job is to redact text deterministically and return it.
  • Be reversible. Return a token map so the caller can rehydrate the original values where appropriate. Expose a matching /unredact endpoint so substitution rules live in one place.
  • Be composable. A chat app, a batch job, a retrieval pipeline, and an eval harness should all be able to call it for their own reasons without pretending to be an LLM caller.
  • Run entirely on-prem. No third-party API calls in the redaction path. Required for HIPAA, PCI-DSS, GDPR-strict, and similar regimes.
  • Be fast enough for interactive use. When called from a chat input on submit (or debounced typing), the redaction round-trip is the user-perceived latency. Target a small fraction of a second.

Non-goals (v1)

  • Proxying LLM calls. The service does not know about LLM backends, OpenAI's schema, auth headers, or streaming responses. See ADR 0002.
  • Server-side session state. Token maps are caller-owned and round-tripped on each call.
  • Multi-tenant auth, rate limiting, audit logging — handled by surrounding infra.
  • Training a custom PII model. We use Presidio's out-of-the-box recognizers plus regex add-ons.
  • Detecting PHI in images, audio, or attached files. Text only.

Components

1. HTTP service

Async HTTP server exposing three endpoints (specified under “API surface” below): POST /redact, POST /unredact, GET /healthz. The service is stateless — no sessions, no caches, no per-caller state. Horizontal scaling is by adding replicas.

2. Redactor

Wraps Presidio's AnalyzerEngine. Responsibilities:

  • Run analysis on the input text.
  • Merge the caller-supplied token map (if any) with new findings, assigning new tokens only for originals not already in the map. This is how cross-call consistency works: the caller passes their accumulated map back in.
  • Apply replacements right-to-left and emit the redacted text + the updated map.

3. Unredactor

Given text and a token map, substitutes tokens back to their originals. Same substitution rules as the redactor uses internally, centralized so callers don't reimplement edge cases (token collision, partial matches).

(Removed) Upstream client

The transparent-proxy design had an “Upstream client” component that forwarded redacted requests to an LLM. ADR 0002 removed it. The service no longer touches LLM backends.

Token-map handling (FAQ)

The token map is { token: original } — e.g. { "[PERSON_1]": "John Doe", "[US_DRIVER_LICENSE_1]": "12345" }. It is the most sensitive object in the system: it contains the raw identifiers. These are the questions adopters ask most about how it is stored and protected. The short version: the service stores nothing; the caller owns the map. This follows directly from the stateless decision in ADR 0002.

Where does the token map live? Who stores it?

The calling application owns and stores it. The map travels in the JSON request/response body, is returned (updated) by /redact, and the caller holds it for the lifetime of the conversation, passing it back on subsequent calls for cross-turn consistency. The service is a pure function per call — it receives the map, uses it, returns it, and retains nothing.

Does the service have a database? Do I provide a connection string?

No, and no. The service has no datastore, no cache, no session store, and no per-caller state. There is no connection string to configure because there is nothing to connect to. If the calling application chooses to persist the map (see below), that is the application's own datastore and the application's own connection string — never the service's.

What storage should the caller use for the map?

Whatever the application already uses for sensitive data, scoped to the lifetime of a conversation:

  • In-memory — fine if the whole conversation is handled within one process/request lifetime.
  • A session store / cache (e.g. Redis) — typical for multi-turn chat; apply encryption at rest and a TTL tied to conversation lifetime.
  • A database (e.g. Postgres) — when conversations must survive restarts; encrypt the column/field at the same trust level as the raw input.

The map should be discarded when the conversation ends; a per-session map also limits the blast radius of the token-cardinality side-channel (threat T10).

Who encrypts it, and where?

  • At rest: the caller's responsibility. The map is exactly as sensitive as the raw input and must be protected at the same trust level. This is threat T6 — a documented caller responsibility the service cannot enforce.
  • In transit: the map crosses the wire between the application and the service, so transport encryption protects it — TLS, or mTLS for the standalone topology. In the sidecar / loopback deployment the map never leaves the host.
  • Inside the service: the map exists only in memory for the duration of a single request and is never written anywhere.

Why isn't storing and encrypting the map the service's job?

Keeping the service stateless buys trivial horizontal scaling (any replica can serve any call), no session-eviction policy, and no shared datastore to breach or threat-model. ADR 0002 explicitly considered and rejected server-held sessions for these reasons. The cost of that choice is that the storage-and-encryption burden moves to the caller — an honest, documented trade.

Could the service offer an optional “hold and encrypt the map for me” mode?

Not in v1. Such a mode would reintroduce server-side state, a datastore, key management, and several threat-model rows that ADR 0002 deliberately removed. It is a legitimate idea, but it reopens that decision and would require a new ADR weighing it — it will not be added silently.

Token format

[<ENTITY_TYPE>_<N>] — e.g. [PERSON_1], [PHONE_NUMBER_2], [DATE_TIME_1].

Chosen so the model:

  • Sees a clear placeholder it can reason about (“the patient is [PERSON_1]”).
  • Can refer back to the same entity consistently.
  • Won't be tempted to invent a value (it sees a non-natural string).

N is allocated against the caller-supplied map, not a per-process counter. If the caller's map already has [PERSON_1] and [PERSON_2], a new person becomes [PERSON_3]. This keeps numbering stable across turns within a conversation and across replicas (the service has no shared state).

Performance approach

Per the v0 notes, we want a hybrid path:

  1. Regex fast path for structured identifiers (SSN, PAN, phone, email, MRN patterns).
  2. NLP slow path (spaCy via Presidio) only for free-text spans that didn't fully match regex.

For v1 we ship with Presidio's default pipeline (which already runs regex recognizers before NER) and measure. The hybrid split becomes a real optimization only if profiling shows the NLP pass dominating.

Because the service is on a caller's hot path, the latency budget is not “added cost on top of an LLM call” (as it would have been in the old proxy design) — it's the whole user-perceived round-trip. See Redaction quality targets.

API surface

POST /redact

Request body:

{
  "text": "Update patient John Doe, MRN 12345.",
  "token_map": {                  // optional; pass back what /redact returned previously
    "[PERSON_1]": "Jane Roe"
  },
  "language": "en"                // optional; defaults to "en"
}

Response body:

{
  "redacted_text": "Update patient [PERSON_2], MRN [US_DRIVER_LICENSE_1].",
  "token_map": {
    "[PERSON_1]": "Jane Roe",
    "[PERSON_2]": "John Doe",
    "[US_DRIVER_LICENSE_1]": "12345"
  }
}

Errors: 400 on malformed body or token-shaped substring in input (see threat T6); 413 if the body exceeds the configured size cap; 500 on engine failure.

POST /unredact

Request body:

{
  "text": "I updated [PERSON_2]'s record.",
  "token_map": {
    "[PERSON_2]": "John Doe"
  }
}

Response body:

{
  "text": "I updated John Doe's record."
}

Behavior on a token in the text that isn't in the map: strict — the redactor core raises UnknownTokenError and the service returns 422, substituting nothing (all-or-nothing). An unknown token means the text and map are out of sync (a corrupted or mismatched map), which a redaction tool should surface loudly rather than silently pass through. (Updated 2026-06-06; supersedes the earlier “left as-is” behavior — see Spec M1-03 and the M2 plan.)

GET /healthz

Returns {"status": "ok"} once the analyzer model is loaded.

Redaction quality targets

The service's only safety promise is that identifiers in the input do not appear in redacted_text in the response. That promise is bounded by the redactor's recall. Precision matters too (over-redaction breaks downstream LLM reasoning) but recall is the hard constraint.

What we measure

  • Per-entity recall — for each entity type the deployment cares about (often the HIPAA Safe Harbor list as a starting set: names, geographic subdivisions smaller than a state, dates more specific than year, phone/fax, email, SSN, MRN, account/license numbers, vehicle/device identifiers, URLs, IPs, biometric identifiers, “any other unique identifying number, characteristic, or code”), the fraction of true positives the redactor catches at the chosen confidence threshold.
  • Per-entity precision — for each entity type, the fraction of redactions that were correct.
  • Span-level F1 — by character offset, not token boundary (a partial match that leaves part of a name visible counts as a miss).
  • End-to-end leakage rate — over a labelled corpus, the fraction of inputs where any entity span escaped redaction. This is what a compliance review will care about.

Targets (per-type — revised 2026-06-07)

The original single “recall ≥ 0.99 across all types” bar (2026-06-05) was retired after the first real gate run: it conflated detection mechanisms with very different achievable ceilings. Targets are now per-type, by mechanism:

TierTypesRecall target
Checksum / formatUS_SSN, CREDIT_CARD, NPI, DEA, EMAIL_ADDRESS≥ 0.99 (deterministic)
Structured patternPHONE_NUMBER, DATE_TIME≥ 0.95
Free-text NERPERSON, LOCATION≥ 0.97 (requires en_core_web_trf)
Context-only IDMRN, MEMBER_ID, RX_NUMBERreport-only (not gated)

Other metrics: per-entity precision ≥ 0.90 (over-redaction hurts UX, not compliance). Latency target (revised 2026-06-07, size-aware): p95 ≤ 50 ms for a typical short (chat-turn) message; large/bulk inputs are slower and, with the transformer model, GPU-bound for low latency. The original single “≤ 50 ms for 500 tokens” bar is met on CPU by neither model (lg 65 ms, trf 437 ms) and is replaced by this size-aware target.

Two honesty caveats

  • Context-only IDs are report-only. MRN, member ID, and Rx number have no checksum or fixed format — they are detectable only via nearby context words. A corpus recall number for them would be a vanity metric, not a production guarantee, so they are measured and reported but not gated.
  • All targets are corpus-relative. They are recall against the synthetic eval corpus — a regression gate, not a production guarantee. Real-world recall depends on how closely the corpus mirrors real inputs (free-form dates, unlabeled IDs, diverse names all lower it).

Leakage

The prior ≤ 1 in 10,000 end-to-end leakage bar is retired as a gate: end-to-end leakage is bounded by the weakest gated type, so with NER types at 0.97 it cannot reach 1/10,000. Leakage is now reported, not gated. Deployments needing a stronger guarantee for free-text identifiers must add compensating controls (human review, egress filtering, the transformer model, or restricting to BAA-covered downstreams).

Model decision & baseline (2026-06-07)

First real gate run on CPU. With en_core_web_lg the gate FAILed — PERSON 0.96, DATE_TIME 0.90, PHONE_NUMBER 0.88. With the transformer en_core_web_trf the gate PASSes — PERSON / DATE_TIME / PHONE_NUMBER all 1.00 (and the report-only context IDs MRN/RX rise to ~1.0), leakage 0.003. Decision: en_core_web_trf is the default model; en_core_web_lg stays available as a faster, lower-recall option. The cost is latency (short-msg p95 ~30 ms — fine; ~500-token ~437 ms on CPU) and image size (torch). This also removes the need for a custom date recognizer or phone tuning. See the M1-08 baseline.

Evaluation corpus

We need a labelled dataset to measure against. Options, in order of effort:

  1. Synthetic prompts covering all target domains (healthcare, finance, legal, GDPR, etc.) with span-annotated identifiers. Cheap to build; coverage matches what we choose to generate.
  2. Public PII/PHI benchmark sets (e.g. i2b2/n2c2 de-identification challenges) under their data-use agreements. Higher signal; data-use review required.
  3. Held-out internal real corpus, sampled from production with proper PII-handling controls. Highest signal; requires a separate data-handling protocol.

Building (1) is a deliverable of its own, and an M1 prerequisite — the quality bar can't be measured without it. Treat the synthetic corpus and its labelling rubric as a separate artifact under evals/.

Methodology notes

  • Compare spans by character offset, not surface form, to avoid being fooled by tokenizer differences.
  • Stratify recall reporting by entity type — overall recall hides catastrophic per-type failures (e.g. 99% overall, 60% on dates).
  • Re-run the full eval on every change to recognizer config or confidence threshold. Treat it as a regression gate, not a one-time benchmark.

Deployment target

Decision (2026-06-05): the primary v1 target is the standalone k8s service below — a shared deployment multiple callers hit — with a docker-compose setup for local dev (per M4). The other topologies remain documented as supported/alternative shapes an adopter may choose.

Candidate topologies

TopologySketchImplications
k8s sidecar to the callerOne prompt-redact container alongside the chat (or other) service in the same pod; caller talks to it over localhostPer-pod scaling, no network hop, no service-mesh policy needed for the call. Bigger pod image. Easy answer for a single dominant caller.
Standalone k8s service (v1 target)prompt-redact deployment fronted by a ClusterIP service; multiple callers share itBest when more than one caller needs redaction (chat + batch + indexer). Adds a network hop; needs mesh-level mTLS / network policy to keep input cleartext off the wire.
VM / bare hostOne or more VMs running the service directlySimplest if there's no orchestrator. Manual scaling, manual rolling deploys.
On-device / edgeRuns at the caller's edge (e.g. a clinical workstation, a laptop)Best latency, hardest deploy and update story; model size is a real constraint at the edge.

Common implications regardless of topology

  • The service is stateless per request, so horizontal scaling is trivial.
  • Cold start is non-trivial — the NER model takes seconds to load. Readiness probe must wait for the model to load, not just for the HTTP socket to open.
  • Container image size is dominated by the model. With the default transformer (en_core_web_trf + torch) expect multiple GB; en_core_web_lg is hundreds of MB. Worth a registry mirror inside the trust boundary (threat T8).

Resolved

Primary v1 target: standalone k8s service (multiple callers share one deployment) + docker-compose for local dev. This drives the M4 deliverables and the supply-chain row (T8) in the threat model. Sidecar / VM / edge remain documented alternatives an adopter may choose.

Alternatives considered

  • Transparent OpenAI-compatible proxy (the original M0 design). Reversed in ADR 0002 — most redactions don't need to go through an LLM, so a proxy is the wrong shape.
  • Cloud DLP (GCP / AWS Macie). Higher accuracy on medical/financial data, managed, but adds network latency and per-request cost, and routes input to a third party. Ruled out by the on-prem requirement.
  • Philter (self-hosted PHI proxy). Purpose-built for the use case but proxy-shaped. Deferred in favor of Presidio for customizability and community.
  • Browser-side NER (e.g. compromise.cool). Reduces network exposure but the client becomes a trust boundary — a tampered client can ship raw input. We keep the redactor as a server-side authoritative call; browser-side scrubbing could be added later as defense-in-depth.

Open questions

All M0 open questions are resolved or explicitly deferred to a named milestone as of 2026-06-05.

  1. Implementation language & topology. Resolved in ADR 0001 (Accepted): sidecar — Go or TypeScript front-end + Python Presidio sidecar. Remaining: final front-end pick (Go vs TypeScript) at M1 start.
  2. Confidence thresholds. Resolved — start at Presidio defaults and tune against the eval corpus during M1 (the recall target ≥ 0.99 is fixed; the threshold is the dial). Presidio returns scores; the cutoff trades recall for precision.
  3. Token-map shape as a long-lived contract. Deferred to M2 — the wire contract hard-binds when the HTTP service ships, so the shape (flat JSON vs versioned envelope vs extension fields) is decided then. Because callers persist the map across calls, its format is a contract we must honor (or version).
  4. Should /redact accept per-call configuration? Resolved — v1 accepts the language hint only; entity allow-lists and custom regexes are out of scope for v1 to keep the surface minimal. Revisit if multi-domain demand appears.
  5. What to do when redaction would empty the input. Resolved — accept and document. A request whose text is only a phone number becomes "[PHONE_NUMBER_1]"; that's expected behavior, not an error.
  6. Deployment target. Resolved — standalone k8s service for v1. See Deployment target above.
  7. Recall/precision targets — confirmed numbers. Resolved — compliance signed off 2026-06-05 on leakage ≤ 1/10,000, recall ≥ 0.99, precision ≥ 0.90, p95 ≤ 50 ms as the bar. See Redaction quality targets.

Dropped from the prior open-questions list after ADR 0002: streaming SSE rehydration (we don't see LLM responses), model-emitted identifiers (same reason), multipart message content (/redact takes plain text).
