Caller integration guide

How an application integrates with prompt-redact. The service redacts text on demand; your application is the orchestrator — it decides when to call, what to do with the redacted text, and it owns the token map. See ARCHITECTURE.html for the deeper design and examples/demo_caller.py for a runnable multi-turn example.

Run it

docker compose up --build      # front-end on :8080, sidecar internal-only
python examples/demo_caller.py # exercises a multi-turn round-trip

API

  • POST /redact — {text, token_map?, language?} → {redacted_text, token_map}
  • POST /unredact — {text, token_map} → {text}
  • GET /healthz — 200 {"status":"ok"} once the model is loaded, else 503

Errors

StatusWhen
400malformed body; input contains a redaction-token-shaped substring (T5); invalid token map; unsupported language
413request body over the size cap (set PROMPT_REDACT_MAX_BODY_BYTES)
422/unredact got a token absent from the map (the text and map are out of sync)
503service not ready (model still loading)
500unexpected engine failure (generic; never echoes your input)

Error responses are deliberately generic — they never echo your input, which may contain PII. To make a failure debuggable anyway, every response carries an X-Correlation-ID header and every error body repeats it as correlation_id. Quote that ID in a support request and an operator can find the matching server-side log line, which records the error type and status only — never your content.

You may set your own X-Correlation-ID (or X-Request-ID) on the request to tie it to your own logs; it is reused as-is when it is log-safe ([A-Za-z0-9._-], up to 128 chars), otherwise the service substitutes its own. The header is echoed back on success and error alike.

The token map — your responsibility

  • Hold it for the conversation. /redact returns the map; store it for the lifetime of the conversation/session.
  • Pass it back on every /redact so a repeated identifier keeps the same token across turns (cross-turn stability — what makes a redacted conversation coherent to a downstream LLM).
  • Call /unredact on any token-bearing text (e.g. an LLM reply) to restore the originals for the user.
  • ⚠️ The token map is the most sensitive object in the system — it literally maps token → original, so it contains the raw identifiers. Treat it at the same trust level as the raw input: encrypt it at rest, never log it, never ship it to a third party. (See token-map handling, threat T6.)
  • Versioning: the current map is the flat {token: original} shape (v1). Any future incompatible format would be a versioned envelope ({"version": …, …}); v1 has no version key, so you can always tell which shape you hold. (See token-map handling.)

When to call /redact

Call it before any downstream use of text that may contain identifiers — sending to an LLM (especially a non-approved one), but also logging, analytics, evaluation, indexing. The value is keeping identifiers out of every downstream system, not just the LLM.

The trust boundary you own (threat T4)

The service only acts when called. It cannot enforce its own use — if your app sends raw text to an LLM without calling /redact first, identifiers leak and the service can't prevent it. This is the caller's contract to honor: code-review every path from user input to a downstream sink, and consider a separate egress control as defense-in-depth.

Input size

prompt-redact targets short, interactive prompts. Large inputs are slower (the NER pass scales with length) and are bounded, not handled: a body over the cap is rejected with 413 rather than hanging. Keep per-call text to chat-prompt size.

Multi-turn pattern (see demo_caller.py)

map = {}
# turn 1
res = POST /redact {text: turn1}              ; map = res.token_map
send res.redacted_text to the LLM (or log/index/…)
# turn 2 — pass the map back
res = POST /redact {text: turn2, token_map: map} ; map = res.token_map   # same id → same token
# rehydrate a reply
user_text = POST /unredact {text: llm_reply, token_map: map}
