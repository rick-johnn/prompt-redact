Spec M1-07 — Eval corpus

Seventh implementation spec for M1. Builds the synthetic, span-annotated corpus the redactor is measured against — the prerequisite for the M1 exit gate (recall ≥ 0.99). The harness that scores against it is the next spec (M1-08).

Package: evals/ — models.py (data model + JSONL IO + validation), generator.py (deterministic generation), corpus/sample.jsonl (a committed sample). Tests: tests/test_eval_corpus.py. No Presidio: the corpus is pure data.

Key decision: generate, don't hand-author

The corpus is generated deterministically from code, not hand-written or pulled from a third-party dataset. Why:

  • Volume & balance. Measuring per-entity recall needs many spans per type; a generator produces them on demand with controllable balance.
  • No third-party data, no network, no new dependency. Matches the project's on-prem / supply-chain posture (threat T8). A stdlib-only seeded generator beats pulling in a faker dependency or a public PHI set with its own data-use review.
  • Reproducible & auditable. A given seed always yields the same corpus, so results are reproducible and the data is regenerable from source rather than an opaque blob.
  • Synthetic only. No real PHI/PII ever enters the repo (CLAUDE.md).

A small sample is committed (evals/corpus/sample.jsonl) for inspection and a load test; the harness regenerates at whatever size it needs.

Data model (models.py)

Example: { id, domain, text, spans: [Span, ...] }
Span:    { start, end, entity_type, value }   # text[start:end] == value

JSONL on disk (one object per line); spans use character offsets so the harness scores by offset, not surface form. validate_example enforces the invariants: each span in-bounds and start < end, the slice equals value, a non-empty type, and no two spans overlap.

Generation (generator.py)

  • Offsets correct by construction. Text is assembled from (chunk, type) parts by _assemble, which tracks the running position and emits a span per typed chunk — a span can never drift from its value.
  • Checksum identifiers minted valid. NPI (Luhn over "80840" + 9), DEA (checksum 7th digit), and credit cards (Visa-style, Luhn-valid) are generated using the same algorithms the recognizers validate, so they are actually detectable. The tests cross-check this against the real is_valid_npi / is_valid_dea.
  • Presidio-canonical type names for built-ins (PERSON, EMAIL_ADDRESS, PHONE_NUMBER, US_SSN, CREDIT_CARD, DATE_TIME, LOCATION) plus the custom NPI, DEA, MRN, MEMBER_ID, RX_NUMBER, so the harness compares predictions to gold by type.
  • Domains: healthcare, finance, pbm, generic — plus a no-PII template (an example with zero spans) so the harness can verify the redactor leaves clean text alone.

Labelling rubric

The rules the generated spans follow (and that any future hand-added example must match), kept here rather than a separate evals/rubric.md to honour the docs-are-HTML convention:

  • Span = the identifier only. Labels/prefixes ("MRN ", "DOB ") are never inside the span; the span is exactly the value a reader would treat as the identifier.
  • PERSON = given + family name, no titles or honorifics.
  • DATE_TIME = the date token as written (e.g. MM/DD/YYYY); surrounding words excluded.
  • Numbers carry no separators beyond their canonical form — SSN as ddd-dd-dddd, phone as (ddd) ddd-dddd, card as 16 contiguous digits.
  • One span per occurrence. A value appearing twice in one text is two spans.
  • No overlaps. If two types could claim the same characters, the corpus picks one (the harness and redactor both assume non-overlapping gold).
  • Offsets are characters, not bytes or tokens; Unicode values are allowed and counted by code point.

On the leakage metric (note for M1-08)

The M0 bar includes end-to-end leakage ≤ 1 in 10,000, which only becomes statistically meaningful at tens of thousands of inputs. The generator can produce that volume (raise n_per_template), but whether M1 gates on leakage at that N or reports it while gating on per-entity recall is a harness decision, deferred to Spec M1-08.

Public surface (evals/)

  • generate_corpus(seed=0, n_per_template=20) -> list[Example]
  • Span, Example, validate_example, CorpusValidationError
  • dumps_jsonl / loads_jsonl / dump_corpus / load_corpus

Test plan (tests/test_eval_corpus.py)

  • Determinism: same seed → identical corpus; different seed → different texts.
  • Offsets: every example passes validate_example; every span slices to its value.
  • Checksums: all NPI/DEA spans pass the real validators; all cards are 16-digit Visa-style and Luhn-valid.
  • Coverage: all expected entity types and domains present; a no-PII example exists; ids unique.
  • JSONL: round-trips; blank lines skipped.
  • Validation: rejects wrong value, out-of-bounds, overlap, empty type.
  • Committed sample loads and validates.

Definition of done

  • evals/ package + committed sample; generation deterministic and offset-exact.
  • test_eval_corpus.py green; full suite green; no Presidio import in evals/.
  • MRN / member ID / Rx are in the corpus as gold even though their recognizers are deferred — so M1-08 can measure them and tune those recognizers in.
