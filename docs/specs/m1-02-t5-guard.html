Spec M1-02 — T5 token-shaped-input guard

Second implementation spec for M1. Implements the threat-T5 guard: reject caller input that already contains a substring shaped like one of our redaction tokens. Pure-Python, builds directly on find_tokens from Spec M1-01 — no Presidio dependency.

Module: prompt_redact_core/guards.py. New error: TokenShapedInputError in errors.py. Maps to the plan's T5 guard note and the test_guards.py test file.

The invariant being protected

The whole redact/unredact contract rests on one invariant:

Every [TYPE_N] substring in redacted text is one this library minted and recorded in the token map.

If a caller's raw input already contains [PERSON_1], that invariant breaks two ways:

  • Redact side: the literal [PERSON_1] passes through into the redacted output, which now contains a token that does not correspond to anything in the map — indistinguishable from a real one.
  • Unredact side: on the round trip, unredact would substitute that stray [PERSON_1] with whatever the map holds for it (or, if the caller minted a real [PERSON_1] elsewhere, corrupt the caller's literal text). Either way the output is wrong.

Rather than try to escape or disambiguate, v1 rejects such input with a typed error. This is documented as a caller constraint (the M2 service maps the error to 400). It is a narrow, well-defined restriction: real-world PII text essentially never contains [UPPER_SNAKE_123] bracketed substrings.

Where the guard runs

  • redact() entry (Spec M1-06): yes. The input is the caller's raw text; it must not already contain token shapes.
  • unredact() entry (Spec M1-03): no. There the input is redacted text and is supposed to be full of tokens. Running the guard there would reject every legitimate call.

So the guard is a redact-path precondition only. This spec ships the guard function; Spec M1-06 wires it into redact().

Grammar: deliberately the minting grammar

The plan sketches the guard pattern as \[[A-Z_]+_\d+\]. This spec instead reuses the canonical TOKEN_RE from Spec M1-01 (\[([A-Z][A-Z_]*)_(\d+)\]) via find_tokens, so the guard and the minter share one definition of "what a token looks like." Rationale:

  • Only strings matching the grammar we actually mint can collide on the round trip. A leading-underscore form like [_X_1] is matched by the plan's broader [A-Z_]+ but is something this library never produces, so it cannot collide — rejecting it would be a false positive against legitimate input.
  • Tying the guard to TOKEN_RE means the guard can never reject a string we would mint, nor admit one we would — the two stay in lockstep if the token grammar ever changes.

Consequence (documented, intentional): lowercase ([person_1]), digit-in-type ([X1_2]), no-number ([PERSON]), and unclosed ([PERSON_1) forms are not rejected, because none of them can be confused with a minted token.

Public surface (guards.py)

  • contains_token_shapes(text) -> bool — non-raising check (does the text contain any minted-token-shaped substring?).
  • assert_no_token_shapes(text) -> None — raises TokenShapedInputError if it does; the error carries every offending match (token + offset) so the M2 layer can build an explanatory 400.

Error (errors.py)

  • TokenShapedInputError(RedactError) — carries .matches (the list of offending TokenMatches) and a human-readable message naming the first offender and its offset.

Test plan (tests/test_guards.py)

  • Accepted (no raise): plain text; empty string (M0: empty input is accepted); bracketed-but-not-token text ([hello], [PERSON]); near-misses that can't collide ([person_1] lowercase, [X1_2] digit-in-type, [PERSON_1 unclosed, [_X_1] leading underscore).
  • Rejected (raise): a bare token; a token embedded in a sentence; multiple tokens; DATE_TIME (underscore type) and multi-digit [PERSON_12].
  • Error payload: .matches length and the first match's token/offset are correct; message names the first offender.
  • contains_token_shapes returns the matching boolean for the accepted/rejected cases.
