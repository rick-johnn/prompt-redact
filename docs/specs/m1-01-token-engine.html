Spec M1-01 — Token engine & map merge

First implementation spec for M1. Covers the pure-Python token vocabulary that everything else in the redactor builds on: the [TYPE_N] format, parsing, the caller-supplied map merge (token assignment), and right-to-left replacement. No Presidio/spaCy dependency — this module is fully unit-testable on its own, which is why it is sequenced first.

Modules: prompt_redact_core/tokens.py, prompt_redact_core/errors.py. Maps to the plan's core-logic section (token format & numbering, map-merge algorithm) and the test_map_merge.py test file.

Token map contract (this spec's working definition)

The caller-owned token map is a flat dict mapping token → original, e.g. {"[PERSON_1]": "John Doe"}. This direction makes unredact (Spec M1-03) a direct lookup and is round-tripped on every call per ADR 0002. The redact path builds the inverse (original → token) index internally so repeated identifiers reuse their token.

Note: the wire-contract shape (flat vs versioned envelope) is formally deferred to M2 (per the M1 plan's out-of-scope list). This spec fixes the in-memory shape only; M2 can wrap it.

Token format

[<ENTITY_TYPE>_<N>] where:

  • ENTITY_TYPE matches [A-Z][A-Z_]* — uppercase letters and underscores only (PERSON, DATE_TIME, US_SSN, CREDIT_CARD, custom MRN/NPI/DEA).
  • N is a positive integer (\d+), allocated per entity type starting at 1.

Why digits are excluded from the type (a deliberate constraint, not an accident): if the type could contain digits, the canonical token regex would be ambiguous. [PERSON_12] could parse as (PERSON, 12) or (PERSON_1, 2), and a greedy matcher resolves it the wrong way. Restricting the type to [A-Z_]+ makes \[([A-Z][A-Z_]*)_(\d+)\] an unambiguous parse. This is exactly the pattern the T5 guard (Spec M1-02) keys on, so the two specs share one definition of "what a token looks like."

Public surface (tokens.py)

  • TOKEN_RE — compiled canonical token pattern.
  • format_token(entity_type, n) -> str — validates the type and n ≥ 1; raises InvalidEntityTypeError / ValueError otherwise.
  • parse_token(s) -> (type, n) | None — exact full-string parse.
  • find_tokens(text) -> list[TokenMatch] — every token occurrence with offsets (the guard and unredactor reuse this).
  • Detection(start, end, entity_type, text) — a frozen value object for a detected PII span (analyzer-agnostic, so the merge is testable without Presidio).
  • Replacement(start, end, token) — a frozen replacement instruction.
  • assign_tokens(detections, token_map) -> (replacements, new_map) — the map-merge algorithm.
  • apply_replacements(text, replacements) -> str — right-to-left substitution.

Map-merge algorithm (assign_tokens)

  1. Validate + index the supplied map. Each key must be a well-formed token (else MalformedTokenMapError). Build the reverse index original → token; if one original is mapped from two different tokens, the map is malformed (the redact-reuse lookup would be ambiguous) → MalformedTokenMapError. Track the highest N seen per type.
  2. Assign, in first-appearance order. Process detections sorted by (start, end) so the lowest N goes to the identifier that appears earliest — stable, human-readable numbering. For each detection: if its text is already in the reverse index, reuse that token; otherwise mint max_N[type] + 1, record it in the new map and the reverse index.
  3. Return the per-detection replacements plus a new map (the input map is not mutated).

Stability guarantees this gives us:

  • Same original → same token within a call and across calls (when the returned map is passed back in) — the cross-call requirement from the plan.
  • Numbering continues from the supplied map: a map already holding [PERSON_2] mints [PERSON_3] for the next new person.
  • Per-type counters are independent.
  • Minting is a function of detection offsets, not input list order — shuffling the detection list yields identical tokens.

Right-to-left replacement (apply_replacements)

Replacements are validated (in-bounds, start ≤ end, non-overlapping — overlaps raise OverlappingSpansError) then applied in descending start order so each splice leaves the offsets of not-yet-applied spans valid.

Errors (errors.py)

  • RedactError — base for everything this library raises.
  • InvalidEntityTypeError(RedactError, ValueError) — entity type doesn't match [A-Z][A-Z_]*.
  • MalformedTokenMapError(RedactError) — supplied map has a non-token key or an ambiguous original.
  • OverlappingSpansError(RedactError) — replacement spans overlap.

Later specs extend this module (e.g. the T5 TokenShapedInputError, the unredact TokenCollisionError).

Out of scope for this spec

  • Detecting PII (that's the analyzer, Spec M1-04) — assign_tokens takes already-detected spans.
  • Case/whitespace normalization of originals — v1 dedupes by exact string match; documented, revisited only if the eval corpus shows it matters.
  • The T5 input guard and unredaction — their own specs (M1-02, M1-03), built on find_tokens.

Test plan

tests/test_tokens.py (format/parse/find/apply) and tests/test_map_merge.py (assignment):

  • format/parse round-trips, including DATE_TIME (underscore in type) and multi-digit N (the [PERSON_12] ambiguity case).
  • format_token rejects lowercase / digit-bearing types and n < 1.
  • find_tokens locates multiple tokens with correct offsets.
  • apply_replacements: single, multiple (R→L correctness), overlap rejection, out-of-bounds rejection.
  • Map merge: empty map mints from 1; two distinct originals number by appearance; repeated original reuses its token; cross-call stability via round-tripped map; numbering continues from a pre-populated map; independent per-type counters; malformed-map rejections (bad key, ambiguous original); minting invariant under input-order shuffle.

Definition of done

  • tokens.py + errors.py implemented; input map never mutated.
  • Both test files green under pytest.
  • No Presidio/spaCy import in either module (keeps the engine independently testable).
