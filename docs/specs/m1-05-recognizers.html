Spec M1-05 — Custom recognizers

Fifth implementation spec for M1 (built after Spec M1-06; the redactor did not depend on it). Adds custom Presidio recognizers for provider identifiers Presidio does not cover out of the box, and registers them with the analyzer (Spec M1-04).

Module: prompt_redact_core/recognizers.py. Touches analyzer.py (new AnalyzerConfig.custom_recognizers flag + registration). Maps to the plan's custom-recognizers scope item and the test_recognizers.py test file.

Scope decision: checksum-validated identifiers only, for now

The M0 corpus decision named five domain identifiers: NPI, DEA, MRN, member ID, Rx number. This spec implements only the two with a self-validating checksum:

  • NPI — 10 digits; the 10th is a Luhn check digit over "80840" + the first 9 digits.
  • DEA — two letters + 7 digits; the 7th digit is a checksum over the first six.

MRN, member ID, and Rx number are deferred to Specs 07-08 (eval corpus + threshold tuning), on purpose. They have no universal format or checksum, so a recognizer for them is a bare numeric/alphanumeric pattern leaning on context words. At the default confidence threshold (M0 chose Presidio defaults, i.e. 0.0) such a pattern fires on every number, which would over-redact catastrophically and tank precision. They can only be calibrated once the corpus exists to measure precision against a tuned threshold — so they ship with that tuning, not before. (Decision 2026-06-05; this keeps the precision bar ≥ 0.90 honest rather than shipping a recognizer we can't yet measure.)

NPI and DEA already deliver the prescriber/provider slice of the payer/PBM theme, and being checksum-gated they are safe to ship without the threshold being tuned.

Design

  • Pure validators. is_valid_npi(text) and is_valid_dea(text) are plain functions with no Presidio import, so they are fully unit-testable on their own. They are the precision-critical logic.
  • Lazy Presidio wrapping. build_custom_recognizers() imports Presidio and defines the PatternRecognizer subclasses inside the function. Subclassing at module level would force a top-level Presidio import; keeping it lazy means importing this module (or the package) never needs the ML stack — the same discipline analyzer.py follows.
  • Checksum via validate_result. Each recognizer overrides Presidio's validate_result hook (the same mechanism the built-in credit-card and SSN recognizers use): a checksum-valid match is confirmed, an invalid one is discarded. So a match survives on the strength of its checksum, independent of the not-yet-tuned threshold. Context words ("npi", "dea", …) are supplied for future score enhancement but are not required for a valid checksum to confirm.
  • Registration. RedactionAnalyzer._build_engine adds the recognizers to the engine registry when config.custom_recognizers is true (default). New entity types: NPI, DEA. They flow through the existing overlap-resolution and Detection conversion unchanged, and compose with the token engine like any other type.

Known precision note

A checksum-valid 10-digit number that is actually something else (≈10% of random 10-digit strings pass the NPI Luhn) would be labelled NPI. The DEA format (2 letters + 7 digits + checksum) is far more specific and rarely collides. Both are revisited when the eval corpus quantifies precision (Specs 07-08); context-weighting or a higher threshold can be applied then if needed.

Public surface (recognizers.py)

  • is_valid_npi(text) -> bool
  • is_valid_dea(text) -> bool
  • build_custom_recognizers() -> list — Presidio PatternRecognizer instances (lazy import).

Change to analyzer.py

  • AnalyzerConfig.custom_recognizers: bool = True.
  • _build_engine registers the custom recognizers when the flag is set.

Test plan (tests/test_recognizers.py)

  • NPI checksum (pure): known-valid NPI accepted; wrong check digit, non-10-digit, non-digit, and empty rejected.
  • DEA checksum (pure): known-valid DEA accepted; wrong checksum, wrong shape (letters/digits count), and empty rejected; case-insensitive letters.
  • Luhn helper: spot-check the check-digit computation.
  • Integration (real Presidio, skipped if unavailable): a valid NPI in context is detected as NPI; a checksum-invalid 10-digit number is not; a valid DEA is detected as DEA; disabling custom_recognizers suppresses them.

Definition of done

  • recognizers.py implemented; analyzer registers NPI/DEA by default.
  • Pure validator tests green; full suite green; branch coverage 100% for the pure paths (excluding integration-only paths).
  • No Presidio import at module load in recognizers.py.
  • MRN / member ID / Rx explicitly carried forward to Specs 07-08.
