Spec M1-09 — Packaging & dev environment

Follow-on to Spec M1-08. The harness is the first code that actually needs Presidio + the spaCy model at runtime, so this spec adds the dependency declaration that makes python -m evals.run_eval runnable, and records the supply-chain decisions.

Files: requirements.in (new), README “Development” section, pyproject.toml comment. No source/test changes.

Decisions

  • requirements.in is the source of truth; requirements.txt is compiled. The direct deps live in requirements.in (presidio-analyzer, spacy, pytest). The install artifact is requirements.txt, produced with pip-compile --generate-hashes requirements.in. Hash-pinning every transitive dependency is the threat-T8 mitigation.
  • requirements.txt is not committed yet. It must be compiled with hashes on a networked machine; committing a hand-written or hash-less version would defeat T8 and rot immediately. It is committed once it can be generated properly. Until then, pip install -r requirements.in gives a working (un-hash-pinned) dev install.
  • The spaCy model is installed separately by default. en_core_web_lg is not on PyPI; the documented path is python -m spacy download en_core_web_lg. For a pinned/offline deploy (T8), the model's release wheel can be added to requirements.in and mirrored inside the trust boundary — noted inline in the file, left out of the default to avoid pinning a version that may not match the resolved spaCy.
  • pytest stays in requirements.in for now (per the plan's dep list) rather than a separate dev-requirements; it can be split out later if a leaner runtime image matters.

Why this isn't verifiable in the design sandbox

The sandbox has no network and no Presidio, so neither pip-compile nor a real run_eval can execute here. This spec therefore delivers the declaration and the instructions; compiling the hash-pinned lockfile and the first gate run are the first actions on a dev/CI machine. The pure tests (token engine, guards, corpus, metrics) remain green here regardless.

Developer workflow (also in the README)

python -m venv .venv && source .venv/bin/activate
pip install -r requirements.in
python -m spacy download en_core_web_lg
pytest                       # integration tests auto-skip without the model
python -m evals.run_eval     # the M1 recall gate

Definition of done

  • requirements.in lists the direct deps with the model + hash-pinning instructions inline.
  • README documents venv setup, model install, test run, and the eval gate.
  • Decisions recorded: compiled+hashed requirements.txt (T8), model install path, why the lockfile isn't committed yet.
