Spec M1-04 — Presidio analyzer wrapper

Fourth implementation spec for M1, and the first to depend on Presidio + spaCy. Wraps Presidio's AnalyzerEngine behind a small, configurable surface that returns the analyzer-agnostic Detection objects from Spec M1-01, so the redactor (Spec M1-06) stays decoupled from Presidio types.

Module: prompt_redact_core/analyzer.py. Maps to the plan's analyzer.py (“AnalyzerEngine wrapper + config: threshold, language, entities”) and the test_analyzer.py file.

Goal

RedactionAnalyzer(config).analyze(text) -> list[Detection]: run Presidio, apply the configured score threshold, resolve overlapping detections deterministically, and return clean, non-overlapping Detections sorted by offset — ready to feed straight into assign_tokens.

🔒 Threat-model hardening: no network egress at analyze time

Empirically (Presidio 2.2.362), the built-in email recognizer calls tldextract.extract(...), whose default extractor attempts to fetch the public-suffix list from publicsuffix.org on first use. In this environment that returned 403 Forbidden; it fell back to the bundled snapshot, so detection still worked — but a PII redaction service must not make outbound network calls while processing input (an air-gapped on-prem deploy would see failed fetches, added latency, and an unexpected egress path). Per CLAUDE.md, PII is the threat model and the service stays self-contained.

Mitigation: when building the engine, the wrapper pins tldextract's module-level extractor to an offline instance (TLDExtract(suffix_list_urls=())) — snapshot only, never fetches. The email recognizer does a module-attribute lookup (tldextract.extract) at call time, so the pin takes effect without touching Presidio internals. This is verified by test: after building, the extractor has no suffix-list URLs, and detection still works fully offline.

Overlap resolution (required)

Presidio returns overlapping results: e.g. for john@example.com it emits both EMAIL_ADDRESS [.,.) score=1.00 and URL [.,.) score=0.50 over the domain. Our right-to-left replacement (apply_replacements) rejects overlaps by design, so the wrapper must pick a non-overlapping subset before handing off.

Algorithm (resolve_overlaps, pure / Presidio-free): sort candidates by (score desc, length desc, start asc, entity-type asc) and greedily keep each one that does not overlap an already-kept span. Highest-confidence wins; ties favor the longer span; remaining ties broken deterministically so the output is stable. Touching spans (end == start) are not overlaps and both survive.

Layering for testability

The conversion and overlap logic operate on a tiny internal value object, ScoredSpan(start, end, entity_type, score), not on Presidio's RecognizerResult. This keeps resolve_overlaps and to_detections pure and fast to unit-test (no model load). analyze() is the only part that touches Presidio: it converts each RecognizerResult to a ScoredSpan, runs the pure pipeline, then converts survivors to Detections (slicing the originating text).

Presidio is imported lazily, inside engine construction — importing analyzer.py (or the top-level package) does not require the ML stack, preserving the pure-import property of Specs 1–3. The analyzer is intentionally not re-exported from the package __init__; callers import prompt_redact_core.analyzer explicitly.

Configuration (AnalyzerConfig)

FieldDefaultMeaning
language"en"The per-call language hint (M0: v1 takes a language hint only).
score_threshold0.0Minimum Presidio score to keep. 0.0 == Presidio defaults (M0 decision); tuned against the corpus in Specs 7–8.
entitiesNoneNone = all supported entity types; otherwise restrict to the given tuple.
spacy_model"en_core_web_lg"The tooling-confirmed model (M1 plan).

Public surface (analyzer.py)

  • ScoredSpan, AnalyzerConfig — value objects.
  • resolve_overlaps(spans) -> list[ScoredSpan], to_detections(spans, text) -> list[Detection] — pure helpers.
  • RedactionAnalyzer(config=None) with .analyze(text) -> list[Detection]; engine is built lazily on first use.

Test plan (tests/test_analyzer.py)

Pure (no model load):

  • AnalyzerConfig defaults.
  • resolve_overlaps: no-overlap passthrough (sorted); higher score wins over a lower-score overlapper; equal score → longer wins; equal score+length → deterministic tie-break; a 3-span chain; touching spans both kept; empty input.
  • to_detections: slices text correctly and sorts by start.

Integration (Presidio-backed, skipped if model unavailable):

  • Detects PERSON, EMAIL_ADDRESS, PHONE_NUMBER in a synthetic sentence; spans slice back to the expected text.
  • Overlap resolution end-to-end: the email/URL overlap yields the EMAIL_ADDRESS, with no URL span covering the same characters.
  • Empty input → [].
  • Threshold filtering: a high score_threshold drops low-score detections but keeps a score-1.0 email.
  • entities restriction returns only the requested type.
  • Offline pin: after building, tldextract.extract.suffix_list_urls == ().
  • Output composes with assign_tokens (the Detection contract holds end-to-end).
