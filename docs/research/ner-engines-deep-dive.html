NER engines deep-dive

Research note, not a decision. Supports ADR 0001 and any future engine-reconsideration ADR. Numbers cited below come from published benchmarks and prior knowledge; treat them as good-enough-to-decide-whether-to-spike, not as production figures. Where exact numbers would lock a decision, the recommendation is to benchmark on the project's own input distribution.

Why this doc exists

The conversation between “Python + Presidio” and “Rust + ONNX” surfaced a comparison (~25–50 ms vs ~5–15 ms p95) that needs interrogation:

  1. Are those numbers credible? What's actually inside each stack?
  2. Is the gap meaningful in this product's call pattern, or is it noise?
  3. Is the choice binary, or is there a sensible “offer both” design?

This document goes deeper into each technology, then re-frames the speed comparison around how the service is actually used.

spaCy

What it is

spaCy is an industrial-strength NLP library built by Explosion AI (Matthew Honnibal and Ines Montani). It has been the default production-grade NLP toolkit in Python since roughly 2016. It is Cython-backed (most hot paths drop to C), opinionated about pipelines, and tuned for batch throughput on CPU.

A spaCy pipeline is a sequence of components — tokenizer, tagger, parser, NER, lemmatizer, attribute ruler — that run in order on each doc. Each component is independently swappable. For redaction, only the tokenizer and the NER component matter; the rest can be disabled to cut latency.

How spaCy NER works

spaCy's CPU-targeted NER uses a transition-based parser with the BILUO tagging scheme (Begin, Inside, Last, Unit, Outside). For each token, a small feed-forward network — trained on top of a bloom-embedding-based feature extractor — predicts the next transition. A beam search keeps the top hypotheses. Compared to a transformer's full attention pass, this is dramatically cheaper per token, at some accuracy cost.

The transformer variant (*_trf models) replaces the cheap feature extractor with a HuggingFace transformer (RoBERTa-based by default). Accuracy goes up; CPU latency goes up substantially.

Memory and concurrency

spaCy's model objects are not safe to share across threads in a way that bypasses the GIL. The practical pattern is multiprocess workers: each worker loads its own copy of the model into RAM. For en_core_web_lg at ~600 MB per worker, an 8-worker deployment uses ~4.8 GB of RAM just for models. At higher concurrency this becomes the dominant cost.

Strengths

  • Mature, well-documented, large community, active commercial backing.
  • Easy to fine-tune on your own labelled data (the docs walk you through it).
  • Pluggable architecture — you can swap the NER component for a transformer pipeline or your own custom component without rewriting the rest.
  • Good baseline accuracy without fine-tuning for common Western entity types.
  • Excellent tokenizer that handles edge cases (URLs, emails inside text, contractions) reliably.

Weaknesses

  • Python-only inference. Latency floor is what Python can do.
  • Each worker loads a full model copy — RAM-expensive at concurrency.
  • The CPU NER tagger's accuracy plateaued years ago; transformer variants get more accuracy but cost much more CPU.
  • Cold start is real — loading en_core_web_lg takes 3–8 seconds. Readiness probes must wait for it.

Presidio

What it is

Microsoft Presidio is an open-source PII detection and anonymization framework, first released in 2019 and actively maintained since. It is not a model — it is a framework that orchestrates detection across multiple kinds of recognizers and provides a clean API for anonymization.

Architecture

Two services that can run together or separately:

  • Presidio Analyzer — finds PII spans in text. Composed of:
    • A recognizer registry with built-in pattern-based recognizers (regex + checksum, e.g. Luhn for credit cards).
    • An NLP engine that does named-entity recognition for free-text entities (PERSON, LOCATION, ORGANIZATION, DATE_TIME). Default backend is spaCy; can be swapped for Stanza, transformers, or a custom backend.
    • A context-aware enhancer that boosts or reduces detection confidence based on surrounding words (e.g. “DOB:” near a date increases its confidence as a date-of-birth).
    • The full pipeline runs all relevant recognizers and merges/dedupes overlapping spans.
  • Presidio Anonymizer — given analyzer results + original text, applies an anonymization operator (replace, redact, mask, hash, encrypt) to each span. Pluggable per entity type.

You can disable recognizers per request (allow-list), pass deny-list contexts, add custom regex recognizers, or register your own recognizer class with arbitrary detection logic. The ad-hoc-recognizer story is good.

What's the Presidio overhead, really?

When you call analyzer.analyze(text), Presidio:

  1. Runs the NLP engine on the text once (spaCy NER + tokenization). This is where most of the time goes — see the spaCy section above.
  2. Runs each pattern-based recognizer's regexes against the text. Regex compilation is cached; matching is fast (sub-ms total even with all built-ins enabled).
  3. Runs validation (e.g. Luhn for credit cards) on regex hits.
  4. Runs the context enhancer, which looks at tokens within a window of each span.
  5. Merges overlapping spans and applies confidence thresholds.

For a 500-token input, Presidio's own dispatch + regex + context layers add roughly 3–5 ms on top of the NLP engine cost. That's the “Presidio tax” — the price of the framework, separate from the model. It's not zero, but it's not the dominant cost.

What Microsoft maintains

Worth being precise here, because it factors into the “I like the Microsoft commitment” reasoning. Microsoft's Presidio team maintains:

  • The recognizer framework and registry.
  • All the pattern-based and US/international recognizers above.
  • The context-aware enhancer logic.
  • The anonymizer with its operator set.
  • Adapters to swap the underlying NLP engine.
  • Documentation, examples, container images, CI.

Microsoft does not maintain:

  • spaCy itself (Explosion AI does).
  • The NER model weights you actually use (also Explosion AI, or HuggingFace if you swap to transformers).
  • The transformer backbone if you swap to a transformers-based NLP engine.

If you go Rust + ONNX, you lose the recognizer framework (which you'd reimplement — a real cost, see below) but you don't lose access to any model. The same spaCy or HuggingFace models can be exported to ONNX and run from Rust.

What reimplementing the Presidio framework actually involves

If you keep the same entity coverage Presidio gives out of the box, you'd write:

  • ~25–30 pattern recognizers, each one a regex and a validation function. Most are 10–30 lines. A competent engineer can do them in ~2–3 days, plus a few days of test coverage.
  • A context-aware enhancer. The Presidio implementation is a few hundred lines; reimplementing the same logic in Rust is straightforward but tedious.
  • A merge/dedupe layer for overlapping spans.
  • An anonymizer with at least the “replace with token” operator (the only one this service needs).

Realistic budget: 1–2 weeks for the framework, plus the corpus and effort to validate that you match Presidio's recall on a test set. Not exotic; not free.

ONNX and ONNX Runtime

What ONNX is

ONNX (Open Neural Network Exchange) is a model-format specification — a way to represent a neural network's computation graph and weights in a portable file. Originally launched by Facebook and Microsoft in 2017, now governed by the Linux Foundation. Most ML frameworks (PyTorch, TensorFlow, scikit-learn, JAX) can export to ONNX.

What ONNX Runtime is

ONNX Runtime is Microsoft-led, open-source inference engine that loads an ONNX model and runs it efficiently across hardware backends. Key properties:

  • Graph optimization. On load, the runtime fuses operators, eliminates redundant nodes, and rewrites subgraphs to use optimized kernels. The optimized graph is what you actually execute.
  • Quantization. int8 quantization of weights and activations typically cuts inference latency 2–4× and model size 4× with a small accuracy hit (often <1% F1 on NER tasks).
  • Cross-language bindings. Official bindings for C++, Python, C#, Java, JavaScript, Objective-C. Community bindings for Rust (ort) are mature and widely used.
  • Hardware backends. CPU (with AVX/AVX-512), CUDA, TensorRT, DirectML, CoreML, ROCm, OpenVINO. Pick at deploy time.

Getting a NER model into ONNX

For HuggingFace transformer-based NER models (e.g. dslim/bert-base-NER, Davlan/distilbert-base-multilingual-cased-ner-hrl), the export path is:

optimum-cli export onnx --model dslim/bert-base-NER --task token-classification ./onnx-out
optimum-cli onnxruntime quantize --onnx_model ./onnx-out -o ./onnx-quantized --avx2

For spaCy models specifically, the export story is rougher — spaCy's CPU tagger is not a transformer and doesn't ONNX-export cleanly. The transformer-based spaCy models (*_trf) export the underlying HuggingFace transformer, but you lose spaCy's beam search and have to reimplement decoding. The honest path for Rust + ONNX is not “export the spaCy model” — it's “pick a HuggingFace NER model that exports cleanly.”

What this means for the comparison

The earlier comparison table implicitly compared different models, not just different runtimes:

  • Python + Presidio path uses en_core_web_lg — spaCy's beam-search tagger.
  • Rust + ONNX path uses something like quantized DistilBERT-NER.

These models have different accuracy and different latency profiles. The fair comparisons are:

  • Python+spaCy lg vs Python+ONNX quantized DistilBERT-NER — isolates the engine swap. Most of the inference speedup is here.
  • Python+ONNX vs Rust+ONNX — isolates the runtime/shell. Rust wins on shell latency, no GIL, RAM efficiency. Inference cost is identical.

Alternative engines worth knowing

Stanza (Stanford NLP Group)

Python NLP toolkit similar in scope to spaCy, with strong multilingual support. NER accuracy is competitive with spaCy lg on English (~89–91% F1). Latency on CPU is generally 1.5–3× slower than spaCy lg. Useful primarily if Stanza's multilingual NER is materially better on a target language than spaCy's.

Flair (Zalando Research)

Python framework with character-level contextual embeddings. State-of-the-art NER F1 on several benchmarks (~93–94%) but CPU latency is high (~100–500 ms per document). Tends to win accuracy benchmarks; loses production deployments to faster options. Worth knowing about; rarely the production answer.

HuggingFace transformers (direct)

Use a fine-tuned NER model directly via the transformers library. Higher accuracy than spaCy CPU models (any well-fine-tuned BERT-base scores ~92–93% F1) but CPU latency is the same order as spaCy *_trf. The natural path here is to export to ONNX for production, not run via PyTorch directly.

GLiNER (2024)

Recent (2024) generalist NER model that achieves near-LLM-level accuracy with a relatively small encoder. Notable because it can do zero-shot entity recognition — give it an entity type as a string, get back spans, no fine-tuning. CPU latency competitive with DistilBERT-NER. Repo. Worth tracking; possibly the future of off-the-shelf NER.

rust-bert

Pure-Rust port of HuggingFace transformers with built-in pipelines for NER, classification, summarization. Uses tch-rs (libtorch bindings) under the hood, so “pure Rust” is slightly misleading — you still have a libtorch dependency. CPU NER latency ~30–80 ms per call. Decent fallback if ONNX Runtime is unavailable.

Custom rule-based (regex + dictionaries)

For the structured entities (SSN, phone, credit card, MRN patterns), pure regex is sub-millisecond and high-recall. For names, locations, and free-text entities, regex alone has terrible recall (a name is just any capitalized word — false-positive nightmare). Custom rules are a great fast path in front of NER, not a replacement.

Speed in context: is 25–50 ms vs 5–15 ms actually meaningful?

The honest answer depends on the call pattern. The same numeric gap is either “completely imperceptible” or “disqualifying” depending on context.

Human perception thresholds (worth grounding against)

LatencyHow it feels to a user
< 100 msInstantaneous. User cannot distinguish 10 ms from 50 ms.
100–300 msNoticeable but feels responsive. UI animations and skeleton states cover it.
300 ms – 1 sSluggish. User starts to question whether anything is happening.
> 1 sBroken-feeling. User context-switches to other tabs.

The same numbers, in different call patterns

Single call per chat submission (the default chat-app pattern)

User hits send → app calls /redact → app calls LLM → app shows response. Redaction is one cost in a chain where the LLM call dominates (hundreds of ms to seconds). 30 ms vs 10 ms is imperceptible in this pattern. The LLM call swamps both.

Debounced per-keystroke (live UI feedback)

User types in an input box; app debounces at 100–200 ms and calls /redact to show what would be sent. The debounce window must accommodate the redaction round-trip. Both 30 ms and 10 ms fit comfortably in a 150 ms debounce; the gap is invisible. Where it'd matter: if you wanted to shrink the debounce window to make the feedback feel snappier (e.g. 50 ms), the 10 ms option keeps you in budget; the 30 ms option doesn't.

Un-debounced per-keystroke

Bad UX pattern (over-calls the service) but worth naming. At a typing speed of ~5 keystrokes/sec, 10 ms keeps you well under the 100 ms perception threshold per keystroke. 30 ms is still under it but uses 30% of the budget. 10 ms wins meaningfully here; 30 ms is borderline.

Batch redaction (audit logs, retrieval indexing, eval corpus prep)

Run /redact across thousands or millions of documents. Per-call latency multiplies directly. 30 ms × 1M docs = 8.3 hours; 10 ms × 1M docs = 2.8 hours. Concrete operational difference.

High concurrency per pod (chat at scale)

Latency per call doesn't change much; what changes is RAM and tail latency:

  • Python: each worker holds ~600 MB of model. 16 workers = 9.6 GB. Tail latency at p99 can balloon to 200–500 ms under GC pressure and GIL contention.
  • Rust+ONNX: one model copy serves all concurrent requests. ~150–300 MB total for a quantized DistilBERT-NER. Tail latency at p99 stays close to median because there's no GC and no GIL.

At 100+ req/s per pod, Rust wins on a different axis than median latency. Median goes from 30 ms to 10 ms (3× better), but p99 goes from 300 ms to 30 ms (10× better) and RAM goes from 10 GB to 300 MB (30× better). The tail is the real story.

Where this leaves the choice

If the service is invoked on a chat hot path with LLM calls downstream, the latency difference is invisible to users. If it's invoked at high concurrency or in batch, the difference is real and operational.

The right question is not “is 25 ms enough better than 50 ms,” it's “what call pattern will dominate, and does the tail matter more than the median?”

What “offering both solutions” actually means

There are four real interpretations of “both,” with quite different trade-offs.

Variant A — Two separate services, same API

Deploy two independent services that both implement /redact and /unredact: one in Python+Presidio, one in Rust+ONNX. Caller picks per route, per environment, or per A/B split.

  • Pros: Maximum isolation. Different teams can own each. Easy to A/B test in production.
  • Cons: Two codebases forever. Two CI pipelines. Two threat models. Two security audits. Two on-call playbooks. Token format and behavior must be kept identical via cross-service tests, or callers can't switch.
  • Honest cost: ~1.7× the maintenance of one service, not 2× — there's shared docs and shared contract — but the duplication is real and permanent.

Variant B — One service, plugin-selectable engine

Build one service (in one language) with a clean engine interface; ship two engine implementations behind it. Selection via env var or config.

  • Pros: Single CI, single deploy artifact, single threat model. Engine swap is a config change, not a re-architecture.
  • Cons: The plugin abstraction is itself code (and a source of bugs). The two engines can't both be written natively — you'd either run Python with both Presidio-in-process and ONNX-in-process (no Rust speedup), or run Rust with both ONNX-in-process and Presidio-as-sidecar (the Python part is still slow). You're choosing one shell language.
  • Honest cost: Less duplication than Variant A. But you don't actually get “both” languages — you get one language with engine choice.

Variant C — Layered hybrid: regex fast path + NER slow path

This is what the existing ARCHITECTURE.html “Performance approach” section already proposes. The fast layer catches structured entities (SSN, phone, email, PAN, MRN) — handles maybe 60–80% of real-world redactions in microseconds. The slow layer runs NER only on inputs that the fast layer didn't fully cover.

  • Pros: Most inputs never hit NER, so “average” latency drops dramatically. Compatible with either Presidio or ONNX (the fast layer is a few hundred lines of regex regardless).
  • Cons: Doesn't help inputs that do contain free-text entities — those still pay the full NER cost. The headline p95 number is essentially unchanged; what changes is the median and the load on the NER engine.
  • Honest cost: One-time engineering investment in the fast path. Worth doing regardless of which engine the slow path uses.

Variant D — Tiered offering: “fast” tier vs “featureful” tier

Productize the choice. The “fast” tier (Rust + ONNX, small entity set, single English model) is for high-throughput / interactive cases. The “featureful” tier (Python + Presidio, full recognizer catalog, multilingual, custom recognizers) is for batch / compliance / multi-jurisdiction cases. Same API, deployed separately, callers pick the URL.

  • Pros: Honest about the trade-off. Different deployment topologies for different use cases without pretending one solution fits both.
  • Cons: Same maintenance cost as Variant A. Marketing/positioning overhead — callers have to understand which tier to pick.
  • When it makes sense: Multiple distinct caller types with genuinely different requirements (e.g. interactive chat and a batch indexer and a multilingual compliance tool).

The honest read on “offer both”

Maintaining two production-grade implementations of the same service is expensive forever. Most teams that go this route end up letting one rot. The cases where “both” pays off:

  • You have two genuinely different caller types with different SLAs (Variant D).
  • You want a migration path from one to the other (Variant A or B, with a clear retirement date for the loser).
  • The fast/slow split is layered within one engine, not two parallel engines (Variant C).

Variant C is the cheapest “both” and it's already in the design. The interesting new question is whether Variants A or D are warranted by an actual second caller type — which depends on M0 inputs we don't yet have (deployment topology, caller mix).

Putting it together

The crisp version of what the deep-dive surfaces:

  1. spaCy is mature and fast-enough for CPU NER, with predictable accuracy and a real RAM cost at concurrency.
  2. Presidio is a thin-ish framework on top of spaCy; the Microsoft commitment is to the framework, not to the model.
  3. ONNX is the portable inference path. Most of the “Rust speedup” really comes from the model swap (transformer NER → quantized small NER), not from Rust itself. Python + ONNX captures most of the inference speedup; Rust + ONNX adds shell latency and concurrency wins on top.
  4. The 25–50 ms vs 5–15 ms gap is invisible to a single chat-app caller. It becomes meaningful at concurrency, in batch, or when measured at p99 instead of p50.
  5. “Offering both” in the strong sense (two parallel implementations) is expensive forever. The cheap version (regex fast path + NER slow path) is already in the design and worth doing.

The decision this informs lives in ADR 0001. The honest next step is a one-day perf spike on the actual input distribution before locking it.

Caveats on the numbers cited above

  • All latency figures come from published benchmarks and prior knowledge, not from running on this project's hardware or input distribution. They are good enough to decide whether to invest in a spike; they are not good enough to lock the decision.
  • spaCy NER F1 figures are on OntoNotes 5, an English news/web-text benchmark. PII recall on clinical, financial, or legal text can be materially different — usually worse on specialized vocabulary.
  • ONNX quantization accuracy loss is “typically < 1%” on standard NER benchmarks. On a fine-tuned domain-specific model, it can be more — verify per model.
  • Concurrency numbers assume a single-pod baseline. Real Kubernetes deployments with HPA and resource limits behave differently.
  • Model F1 is not the same as PII leakage rate. A 91% F1 model can still miss the wrong things — what matters is per-entity-type recall on the entities you care about. See ARCHITECTURE — Redaction quality targets.
