M1 — Redactor core (current)

Detailed plan: plans/m1-redactor-core.html. Specific tooling (dependency manager, spaCy model) is being confirmed there.

Goal: a standalone redactor that takes text + an existing token map, returns redacted text and an updated map. No HTTP, no service yet.

  • Presidio analyzer + anonymization that emits [TYPE_N] tokens.
  • Caller-supplied map merge: existing originals reuse their tokens; new originals mint new tokens against the supplied map.
  • Unit tests covering:
    • Names, phone numbers, emails, dates, addresses, MRN-shaped IDs, financial PANs.
    • Repeated identifiers across multiple calls (with the map passed back in) get the same token.
    • Empty / all-PII / no-PII inputs.
    • Inputs that contain token-shaped substrings are rejected (threat T5).

Exit: tests green; redactor round-trips a synthetic multi-domain corpus with per-entity recall ≥ 0.99 (the bar confirmed in M0).

M2 — Redaction service MVP

Detailed plan: plans/m2-service.html (draft — sidecar topology, with the deferred front-end/IPC/wire-contract decisions called out).

Goal: a runnable service exposing the API in ARCHITECTURE.html — API surface.

  • POST /redact — accepts {text, token_map?, language?}; returns {redacted_text, token_map}.
  • POST /unredact — accepts {text, token_map}; returns {text}.
  • GET /healthz — returns ok once the analyzer model is loaded.
  • Request size cap configurable via env var (threat T9).
  • Integration test: a synthetic caller sends a multi-turn conversation through /redact + /unredact, asserting cross-turn token stability and round-trip fidelity.

Exit: curl against the service round-trips text through /redact and /unredact with stable tokens across calls; structured logs contain no raw identifiers.

M3 — Performance: hybrid regex + NER

Only do this if M2 profiling shows the NER pass dominates p95 latency.

  • Pre-filter input with a fast regex pass for structured identifiers (SSN, PAN, phone, email).
  • Only invoke the spaCy pipeline on spans that aren't fully covered by regex.
  • Bench harness comparing pre- and post-optimization on a fixed corpus.

Exit: documented p50/p95 latency improvement on the bench corpus; quality metrics unchanged.

M4 — Deployment

  • Dockerfile (multi-stage, slim runtime).
  • docker-compose.yml for local dev: the service plus a tiny demo caller that exercises /redact and /unredact.
  • Readiness probe that waits for the spaCy model to load.
  • Documentation on caller integration: when to call /redact, how to store the token map, the trust boundary established by threat T4.

Exit: docker compose up brings up the stack; the M2 integration test passes against the composed stack.

Open questions

These block specific milestones rather than the plan as a whole:

  • Final front-end language (Go vs TypeScript; blocks M1 start) — topology decided in ADR 0001 (sidecar + Python Presidio); the front-end pick is made at the start of M1.
  • Confidence thresholds (blocks M1 exit) — recall target confirmed (≥ 0.99); the threshold itself is tuned against the eval corpus during M1, starting from Presidio defaults. See ARCHITECTURE.html open question #1.
  • Token-map shape as a long-lived contract (deferred to M2 — the wire contract hard-binds when the service ships) — see ARCHITECTURE.html open question #2.

Deferred / out of scope

  • Transparent OpenAI-compatible proxy mode. See ADR 0002 for why.
  • Multi-tenant auth, rate limiting, audit logging — handled by surrounding infra.
  • Streaming responses, multipart content. Not in scope after the service-shape pivot.
  • Server-held sessions or token-store. Maps are caller-owned per ADR 0002.

Open questions resolved: per-call config = language hint only; confidence threshold = Presidio defaults tuned in M1; empty-input = accept and document; token-map contract deferred to M2.
  • 2026-06-07 — Quality targets revised to per-type after the first real gate run (en_core_web_lg). The single recall ≥ 0.99 bar is replaced by per-mechanism tiers: checksum/format ≥ 0.99; structured pattern (phone, date) ≥ 0.95; free-text NER (person, location) ≥ 0.97 (needs en_core_web_trf); context-only IDs (MRN, member ID, Rx) report-only. The ≤ 1/10,000 leakage bar is retired as a gate (reported only). Targets are corpus-relative (a regression gate, not a production guarantee). See ARCHITECTURE — quality targets.
  • 2026-06-07 — Adopted en_core_web_trf as the default model. The gate run showed lg fails (PERSON 0.96, DATE 0.90, PHONE 0.88) while trf passes every per-type target (PERSON/DATE/PHONE → 1.0; leakage 0.003) — so no custom date recognizer or phone tuning is needed. lg stays a configurable fast/low-recall alternative. Latency target made size-aware (p95 ≤ 50 ms for short messages; trf ~30 ms; large inputs GPU-bound). Adds spacy-transformers + torch (larger image).
