M1 Plan — Redactor core

Detailed plan for the M1 milestone. M1 builds a standalone Python redactor library — no HTTP, no service, no front-end (those are M2). The deliverable is the core logic that turns text + a caller-supplied token map into redacted text + an updated map, plus the synthetic eval corpus and harness needed to prove it meets the quality bar locked in M0.

Status: Ready — tooling confirmed (2026-06-05). Per CLAUDE.md this is a design deliverable; no source files are created until “build” is signalled.

Goal

A library function redact(text, token_map) -> (redacted_text, token_map) that:

  • detects PII spans via Presidio (Presidio's default pipeline already runs regex recognizers before NER);
  • assigns [TYPE_N] tokens, reusing tokens for originals already present in the supplied map and minting new ones otherwise;
  • returns the redacted text and the merged map.

Plus the matching unredact(text, token_map) -> text substitution (see Scope for why it's in M1), the eval corpus, the eval harness, and the test suite.

Scope

In scope

  • Pure-Python redactor core (importable library; no network surface).
  • redact() — analyze → token assignment + caller-map merge → right-to-left replacement.
  • Custom Presidio recognizers for domain identifiers not covered out of the box — MRN, payer/PBM member IDs, Rx numbers, prescriber NPI/DEA.
  • unredact() — centralized token → original substitution. Included in M1 because the exit criterion is a round-trip over the corpus, and centralizing substitution rules (collision, partial matches) here keeps M2's /unredact a thin wrapper.
  • Token-shaped-substring guard (threat T5) — raise a typed error the M2 service will map to 400.
  • Synthetic, span-annotated eval corpus under evals/ (a deliverable of its own; prerequisite for measuring the bar).
  • Eval harness computing the M0 metrics; run as a regression gate.
  • Unit tests (see Test plan).

Out of scope (deferred to M2+)

  • HTTP service, the /redact / /unredact / /healthz endpoints (M2).
  • The Go/TypeScript front-end and the loopback IPC boundary (M2; see ADR 0001).
  • Dockerfile / docker-compose / k8s manifests (M4).
  • Per-call entity allow-lists / custom regexes — v1 takes the language hint only (M0 decision).
  • Token-map wire-contract shape (flat vs versioned envelope) — deferred to M2.

Prerequisite: synthetic eval corpus

The quality bar cannot be measured without a labelled dataset, so the corpus is built first (or alongside the redactor). Synthetic data only — no real PHI/PII in the repo, per CLAUDE.md.

Format (proposed)

JSONL, one record per input. Spans are character offsets so scoring is offset-based, not surface-form-based.

{
  "id": "health-0001",
  "domain": "healthcare",
  "text": "Update patient John Doe, MRN 12345, seen 03/04/2021.",
  "spans": [
    {"start": 15, "end": 23, "type": "PERSON",    "value": "John Doe"},
    {"start": 29, "end": 34, "type": "MRN",       "value": "12345"},
    {"start": 41, "end": 51, "type": "DATE_TIME", "value": "03/04/2021"}
  ]
}

Domain & entity coverage

Baseline entity set = HIPAA Safe Harbor types (names, geographic subdivisions, dates finer than year, phone/fax, email, SSN, MRN, account/license numbers, vehicle/device IDs, URLs, IPs, etc.), generated across multiple domains so per-type recall can be stratified:

  • healthcare (names, DOB, MRN, dates) — HIPAA Safe Harbor baseline
  • finance (account numbers, card PANs, SSN)
  • payer / PBM identifiers (member IDs, Rx numbers, prescriber NPI/DEA) — require custom recognizers; not covered by Presidio out of the box
  • generic / GDPR PII (emails, phones, addresses, IPs, URLs)
  • adversarial cases: empty input, all-PII input, no-PII input, and inputs containing token-shaped substrings (for T5).

Legal (client names, matter numbers) is a candidate for a later corpus expansion, not v1.

Labelling rubric

A short written rubric defining what counts as a span boundary per type (e.g. does a name include a title? does a date include the weekday?). Stored alongside the corpus so labels are reproducible and reviewable.

Core logic details

Token format & numbering

[<ENTITY_TYPE>_<N>]. N is allocated against the caller-supplied map, not a per-process counter — so numbering is stable across calls and across replicas (the core holds no state). If the map already has [PERSON_1]/[PERSON_2], a new person becomes [PERSON_3].

Map-merge algorithm

  1. Build a reverse index original → token from the supplied map.
  2. For each detected span: if its original is already in the reverse index, reuse that token; else mint the next N for that entity type and add it to the map.
  3. Apply replacements right-to-left (descending start offset) so earlier offsets stay valid as the string mutates.

Confidence threshold

Start at Presidio defaults (M0 decision); expose the threshold as config; tune it against the corpus during M1, trading recall vs precision. Re-run the full eval on every threshold/recognizer change (regression gate).

T5 guard

Pre-scan input for the pattern \[[A-Z_]+_\d+\]. On a hit, raise a typed error (M2 maps it to 400 with an explanatory message). Documented as a caller constraint.

Eval harness & metrics

Metrics, computed by run_eval.py over the corpus (definitions per ARCHITECTURE — quality targets):

  • Per-entity recall and per-entity precision, stratified by type (overall numbers hide per-type failures).
  • Span-level F1 by character offset (a partial match that leaves part of a name visible counts as a miss).
  • End-to-end leakage rate — fraction of inputs where any labelled span escaped redaction. This is the compliance-facing number.

The harness compares spans by offset (not surface form) and exits non-zero if any gate is missed, so it can run in CI.

Test plan

  • Detection across types: names, phone numbers, emails, dates, addresses, MRN-shaped IDs, financial PANs.
  • Cross-call token stability: the same identifier, with the map passed back in, gets the same token across calls.
  • Edge cases: empty input, all-PII input, no-PII input.
  • T5: inputs containing token-shaped substrings are rejected.
  • Round-trip fidelity: unredact(redact(text)) reconstructs the original for the corpus.

Exit criteria

  • Test suite green.
  • On the synthetic corpus: per-entity recall ≥ 0.99 and end-to-end leakage ≤ 1/10,000 (the M0 bar).
  • unredact(redact(text)) round-trips the corpus losslessly.
  • Eval harness runnable as a regression gate.

Latency (p95 ≤ 50 ms) is validated more meaningfully in M2/M3 under the service; M1 records a baseline.

Sequencing

  1. Confirm tooling (below).
  2. Build the synthetic corpus + labelling rubric (small first slice, expand).
  3. Stand up the eval harness against the corpus (red, since no redactor yet).
  4. Implement analyzer.py + tokens.py + redactor.py; iterate threshold to green the recall gate.
  5. Implement unredactor.py + guards.py.
  6. Fill out the test suite; wire the eval harness as a regression gate.
