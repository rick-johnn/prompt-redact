M4 Plan — Deployment / packaging

Plan for the M4 milestone: package the two-process sidecar (Go front-end + Python sidecar) so docker compose up brings up the stack and the M2 multi-turn integration test passes against it.

Status: Draft, updated after architecture-team review (Petzold, Box, Carmack — full comments in M4_PACKAGING_DISCUSSION.md). The review's throughline: workload is the gate (every packaging lean is downstream of it), and measure before deciding — most size/latency figures here are estimates that one real build + load test turns into facts. Verification gap: Docker is unavailable in the design sandbox, so artifacts are authored here and built/verified on a real Docker host. No Dockerfiles are authored until the prerequisites below are answered.

Decisions resolved (2026-06-08 — workload = interactive chat prompts)

The gating question is answered: the workload is short, interactive prompts in AI chat clients — not large documents or bulk batches. That collapses most of the open decisions:

  • CPU only — no GPU variant. Short turns measure ~20 ms p95 on CPU trf (well under the 50 ms target); GPU was only justified by bulk latency.
  • Ship trf only. The lg variant's case was size + bulk-latency; bulk is gone and trf is already fast enough, leaving only a ~1.5 GB saving for an image that fails the recall gate. lg stays a documented --build-arg escape hatch, not a shipped/tested artifact (so no runtime model-env knob needed).
  • Single synchronous /redact — no async/batch verb (Box's interface question): correct for a chat turn.
  • Bound large pastes, don't handle them. A user can still paste a huge prompt; the request size cap (T9) rejects over-limit input with 413 rather than hanging — set the cap to a sensible chat-prompt limit.
  • Token-map wire contract stays the M2 decision: flat {token: original}, documented v1 (Box's versioning concern is answered).

Net: M4 is one CPU trf sidecar image + the Go front-end + compose + the security tests. No GPU, no variants, no batch verb, no model knob.

Goal (M4 exit)

docker compose up brings up the front-end + sidecar; both report healthy; the M2 conversation test passes when pointed at the composed stack. Caller-integration docs explain when to call /redact and how to hold the token map.

Prerequisites (before any Dockerfile) — from the review

  1. Answer the workload question. Short interactive turns vs bulk/large documents gates the image, the model, and CPU-vs-GPU. Get the load profile from whoever owns the calling app before deciding the rest. (All three reviewers; Carmack: “it is everything.”)
  2. Measure, don't estimate. Run the throwaway build in measure/ on the Docker host to replace the size/latency tildes with real numbers — in particular build with the CPU-only torch wheel (most of the 1–2 GB is CUDA the CPU path never runs; this likely deflates the image-size tension and the CI-RAM problem at once). Capture real image size, cold-start, and p95/p99 latency on the actual deploy hardware.

From measure/ — real numbers replacing the estimates:

What the numbers change:

  • Image size is real even CPU-only. trf = 3.91 GB single-stage with the CPU-only torch wheel — CPU-only is the right default but did not shrink it dramatically (torch + the transformer model are the bulk, not CUDA). The multi-stage production image (M4-01) trims this to 3.01 GB (no build tools/pip cache in the final layer). Still multi-GB; the tension stands but is eased.
  • The lg variant, built right, is 2.35 GB. The naive shared build was 3.67 GB (it installs torch + spacy-transformers, which lg doesn't use). A model-split lg image (no torch/transformers — measure/Dockerfile.lg) is 2.35 GB — real, and bigger than the ~1–1.5 GB first guessed (lg's word-vector model + the spaCy/SciPy/presidio stack are ~2 GB on their own). So lg saves ~1.5 GB vs trf (3.91 → 2.35), a ~40% cut — but both are still multi-GB and need a registry mirror regardless. Implication: the size argument for an lg variant is weaker than hoped; its stronger case is latency on bulk inputs, so the variant decision leans even harder on the workload question. M4-01 needs model-specific deps either way (lg ≠ trf).
  • Cold start is a non-issue (~4 s, not tens of seconds) — the readiness worry largely dissolves.
  • Interactive latency is fine (~20 ms); large-document latency (~470 ms) confirms the bulk-workload concern.

Scope

In

  • Sidecar image — Python + Presidio + the spaCy model, uvicorn entrypoint, model loaded at build (offline at runtime).
  • Front-end image — multi-stage Go build to a tiny runtime image.
  • compose — front-end published; sidecar on the internal network / loopback only, never published (T11). Readiness wired so the front-end depends on the sidecar being model-loaded.
  • Hash-pinned requirements.txt — compiled in the build environment (Python 3.11), used by the sidecar image (closes the deferred compile).
  • Caller-integration docs — the trust boundary (T4), token-map handling, when to call.

Out

  • k8s manifests — the standalone-k8s target is the documented v1 deployment, but compose is the M4 deliverable; manifests can follow.
  • CI pipeline (incl. the memory needed for trf integration tests — see note).
  • GPU image as the default (offered as a documented variant).

Decisions to work through (workload first)

  1. Primary workload (the gate). Interactive short turns or bulk/large documents? Measured latency jumps ~14× from a short message to 500 tokens on CPU trf, so “large document” is the same path scaled until it hits a timeout — that ceiling must be a documented limit. If bulk is real, it may warrant its own async/batch verb (see interface decisions).
  2. Which image(s) to ship. trf-only (meets the bar; one artifact to sign/scan) vs trf + lg variants. If two ship, they must be interface-identical (same token types for the same input; recall-only differs) — else a caller's token map won't round-trip across whichever is deployed, and they're two services. The recurring cost of two variants is the doubled quality-test matrix, not the second Dockerfile. Unresolved pending workload + real lg demand; a documented one-line --build-arg override may cover lg without a second official artifact.
  3. Model bundling. Bake in at build (no runtime network — required for air-gap; reproducible only if the model is pinned by version + digest, like the deps) vs download-at-start (non-starter for air-gap) vs volume mount. Lean: bake in, pinned by hash, in its own cache-friendly layer.
  4. CPU vs GPU. CPU default (ship the CPU-only torch wheel) + a genuinely separate GPU image (CUDA wheel) for bulk/large inputs. Lean: CPU default + documented GPU variant — confirm against the workload.
  5. Base images. python:3.11-slim sidecar; distroless-static Go front-end. Pin the build platform (--platform linux/amd64) so arch doesn't depend on whose laptop built it (an arm64 image fails to start on x86_64). Multi-arch only when a real adopter needs it.

Interface decisions M4 must not foreclose silently (Box)

  1. Token-map schema versioning. The flat {token: original} map is the contract every integrator couples to. Decide whether/how it's versioned before adding entity types changes it under callers.
  2. One verb for all workloads? A 437 ms synchronous /redact plus bulk batches over the same endpoint is a timeout waiting to happen. If bulk is real, decide whether it needs an async/batch verb — cheap now, expensive to retrofit.

Components

  • Sidecar Dockerfile — python:3.11-slim; install hash-pinned reqs; python -m spacy download en_core_web_trf (or pinned wheel) at build; run uvicorn prompt_redact_service.app:app; HEALTHCHECK on /healthz; bind to the internal network only.
  • Front-end Dockerfile — golang:1.22 build stage → distroless static; runs the proxy; env points PROMPT_REDACT_UPSTREAM at the sidecar service name.
  • compose — two services; only the front-end port is published; depends_on with a healthcheck condition so traffic isn't routed before the model loads; a demo/caller note for the round-trip.

Readiness

The sidecar's model load takes ~4 s (trf, measured 2026-06-07 — far less than the feared tens of seconds); /healthz already flips ready only after warmup (M2-01). Compose must gate the front-end on depends_on: condition: service_healthy — a generous start_period alone only delays failure-counting, it doesn't hold traffic. Define the front-end's behavior during the warmup window (fast-fail with a clear 503 rather than hang or queue), since every crash-loop, rolling deploy, and OOM-kill pays the load cost again.

Exit criteria

  • docker compose up → both services healthy (front-end gates on the sidecar via depends_on: condition: service_healthy, not mere container start).
  • The M2 conversation round-trip passes against the front-end's published port — the demo caller is a first-class, CI-green artifact (the executable spec of the contract).
  • Tested security properties (not asserted captions): an automated check confirms the sidecar is unreachable from outside the compose network (negative test), and a test fails if any request body reaches a log sink (no-PII-logging, T2).
  • Real measured numbers recorded (image size, cold-start, p95/p99 latency on deploy hardware), replacing the estimates in this plan.

Note: CI memory for the trf integration tests

Running the full Python suite loads the transformer model across several integration fixtures and can exhaust memory on a small runner (observed: OOM in the sandbox). CI for the integration tests needs adequate RAM, or they should be run sharded / in separate processes. Pure (non-integration) tests are unaffected.
