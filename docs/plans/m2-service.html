M2 Plan — Redaction service MVP

Detailed plan for the M2 milestone: wrap the M1 redactor core (prompt_redact_core) in a runnable HTTP service exposing the API surface — POST /redact, POST /unredact, GET /healthz.

Status: Decisions resolved (2026-06-06) for the MVP path — see Decisions. M2 ships the Python service as the MVP (FastAPI); the Go/TS front-end and IPC protocol are deferred to a later front-end spec. No service code is written until “build” is signalled.

Topology (from ADR 0001)

The accepted shape is a sidecar: a fast front-end (Go or TypeScript) owns the public HTTP API and calls a Python sidecar — which wraps prompt_redact_core — over a loopback-only IPC boundary (threat T11).

   client ──HTTP──▶ ┌────────────────────┐  loopback   ┌────────────────────────┐
                    │ front-end (Go/TS)  │ ──IPC──────▶ │ Python sidecar         │
                    │ - API, size cap,   │             │ - prompt_redact_core   │
                    │   error mapping    │ ◀────────── │ - Presidio analyzer    │
                    └────────────────────┘             │   (built once, eager)  │
                                                        └────────────────────────┘
                         one host / one pod; IPC never leaves the host

MVP scope (decided 2026-06-06): M2 ships the Python service alone — it is a complete redaction service on its own. The Go/TS front-end and the loopback IPC are added in a later spec (the public API and the sidecar's API are the same contract, so the front-end is a thin pass-through). This is why the Go-vs-TS and IPC-protocol picks stay deferred.

Goal

A docker compose up-able service where a synthetic caller can round-trip a multi-turn conversation through /redact + /unredact with stable cross-turn tokens, and structured logs contain no raw identifiers.

Scope

In scope

  • Python FastAPI service exposing /redact, /unredact, /healthz, building the analyzer eagerly at startup (the analyzer's own note: lazy first-build isn't synchronized, so the service must warm it before serving concurrency).
  • Request size cap (T9) and error mapping in the service.
  • The Go/TS front-end + loopback IPC are deferred to a later spec (M2-05) — the Python service is the MVP.
  • GET /healthz readiness — returns ok only once the model is loaded.
  • Structured logging with no request/response bodies by default (T2/T3).
  • Integration test: a multi-turn conversation through both endpoints asserting cross-turn token stability and round-trip fidelity.

Out of scope (later milestones)

  • Dockerfile / docker-compose / k8s manifests — packaged in M4 (compose may be pulled forward for the integration test; TBD below).
  • Auth, rate limiting, multi-tenant concerns — surrounding infra (ADR 0002 non-goals).
  • Per-call entity config beyond the language hint (M0 decision).
  • The deferred MRN/member-ID/Rx recognizers and their tuning (M1 follow-up).

Decisions (resolved 2026-06-06)

  1. Sequencing — Python service first (MVP). The Python service speaking HTTP is a complete redaction service; M2 ships it as the MVP and adds the Go/TS front-end later. This defers decisions #2 and #3.
  2. Front-end language (Go vs TypeScript) — DEFERRED to the front-end spec (M2-05); not needed for the Python MVP.
  3. IPC protocol (HTTP vs gRPC) — DEFERRED to the front-end spec; not needed until there are two processes.
  4. Python web framework — FastAPI. Request validation + OpenAPI out of the box; rejects malformed bodies early (good for a PII API). Adds pydantic + starlette + uvicorn to the deps.
  5. Token-map wire contract — flat dict, documented v1. The wire shape is the in-memory shape, {token: original}, documented as v1. Versioning is added only if a breaking change is ever needed.
  6. /unredact unknown token — strict, mapped to 422. Keep the M1 core's strict behavior (raises UnknownTokenError); the service returns 422. An unknown token signals a corrupted/mismatched map and is surfaced loudly. ARCHITECTURE is updated to match (its prior “leave as-is” wording is superseded).

Components

Python service (FastAPI) — the M2 MVP

  • Builds one RedactionAnalyzer at startup; /healthz flips ready after warmup.
  • redact: calls prompt_redact_core.redact(text, token_map, analyzer=…); maps TokenShapedInputError/RedactError to 400.
  • unredact: calls prompt_redact_core.unredact; maps UnknownTokenError to 422.
  • Request size cap → 413 (T9); structured logging without bodies (T2/T3).

Front-end (Go/TS) — deferred

  • Added in a later spec (M2-05): the fast public shell + loopback IPC client. Because the front-end and the sidecar share the same API contract, it is a thin pass-through. Needs decisions #2 (Go vs TS) and #3 (IPC) first.

API surface

Exactly as in ARCHITECTURE — API surface. Error mapping: malformed body or token-shaped input → 400; body over the cap → 413; unknown-token on unredact → 422; engine failure → 500.

Threats addressed

  • T2/T3 — no request/response bodies in logs; crash handlers scrub.
  • T9 — configurable request-size cap at the front-end.
  • T11 — IPC bound to loopback; mTLS only if ever split across hosts.
  • T4 — documented boundary: the service can't force callers to use it.

Exit criteria (from PLAN)

  • curl round-trips text through /redact and /unredact with stable tokens across calls.
  • Structured logs contain no raw identifiers.
  • Integration test: multi-turn conversation asserts cross-turn token stability + round-trip fidelity.

Spec breakdown

  1. M2-01 — FastAPI app skeleton + eager analyzer build + /healthz readiness; add FastAPI/uvicorn to requirements.in.
  2. M2-02 — /redact + /unredact handlers, flat-v1 wire contract, error mapping (400 / 422 / 500).
  3. M2-03 — request size cap (413, T9) + logging discipline (no bodies, T2/T3).
  4. M2-04 — multi-turn integration test (cross-turn token stability + round-trip); compose may be pulled forward to run it.
  5. M2-05 (later) — Go/TS front-end + loopback IPC; requires decisions #2 and #3.
