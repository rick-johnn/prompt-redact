M2 Plan — Redaction service MVP

Detailed plan for the M2 milestone: wrap the M1 redactor core (prompt_redact_core) in a runnable HTTP service exposing the API surface — POST /redact, POST /unredact, GET /healthz.

Status: Draft / sketch — several decisions deferred to M2 are now due and are called out under Decisions to make. No service code is written until they're settled and “build” is signalled.

Topology (from ADR 0001)

The accepted shape is a sidecar: a fast front-end (Go or TypeScript) owns the public HTTP API and calls a Python sidecar — which wraps prompt_redact_core — over a loopback-only IPC boundary (threat T11).

   client ──HTTP──▶ ┌────────────────────┐  loopback   ┌────────────────────────┐
                    │ front-end (Go/TS)  │ ──IPC──────▶ │ Python sidecar         │
                    │ - API, size cap,   │             │ - prompt_redact_core   │
                    │   error mapping    │ ◀────────── │ - Presidio analyzer    │
                    └────────────────────┘             │   (built once, eager)  │
                                                        └────────────────────────┘
                         one host / one pod; IPC never leaves the host

Goal

A docker compose up-able service where a synthetic caller can round-trip a multi-turn conversation through /redact + /unredact with stable cross-turn tokens, and structured logs contain no raw identifiers.

Scope

In scope

  • Python sidecar: a thin web app exposing redact/unredact/health over loopback, building the analyzer eagerly at startup (the analyzer's own note: lazy first-build isn't synchronized, so the service must warm it before serving concurrency).
  • Front-end: the public HTTP API, request size cap (T9), error mapping, and the IPC client.
  • GET /healthz readiness — returns ok only once the model is loaded.
  • Structured logging with no request/response bodies by default (T2/T3).
  • Integration test: a multi-turn conversation through both endpoints asserting cross-turn token stability and round-trip fidelity.

Out of scope (later milestones)

  • Dockerfile / docker-compose / k8s manifests — packaged in M4 (compose may be pulled forward for the integration test; TBD below).
  • Auth, rate limiting, multi-tenant concerns — surrounding infra (ADR 0002 non-goals).
  • Per-call entity config beyond the language hint (M0 decision).
  • The deferred MRN/member-ID/Rx recognizers and their tuning (M1 follow-up).

Decisions to make (deferred to M2 — now due)

  1. Sequencing: Python-service-first, or full sidecar at once? The Python sidecar is needed regardless and, if it speaks HTTP, is itself a working redaction service. Recommended: ship the Python service as the M2 MVP (proves the API + round-trip), then add the Go/TS front-end as a follow-up. This de-risks M2 and matches ADR 0001's own “honest take” that the front-end shell is not the bottleneck. The alternative is building both for M2.
  2. Front-end language: Go vs TypeScript. The pick ADR 0001 deferred (it nominally said “at M1 start,” but M1 was pure Python). Needed once we build the front-end. Drivers: team familiarity, the surrounding stack, on-call ergonomics.
  3. IPC protocol: HTTP-over-loopback vs gRPC. HTTP is simplest and reuses the same JSON contract; gRPC adds a schema + codegen but is faster and strongly typed. For loopback chat-rate traffic, HTTP is likely enough. Must stay loopback-only (T11).
  4. Python web framework. FastAPI vs Starlette vs Litestar (ADR 0001's Option-A candidates). FastAPI gives request validation + OpenAPI cheaply.
  5. Token-map wire contract (deferred from M0). The in-memory shape is a flat {token: original} dict. Decide the JSON wire shape now: flat dict, or a versioned envelope ({"version":1, "map":{…}}) for future-proofing. Callers persist it, so it's a contract to honor or version.
  6. /unredact on an unknown token. ARCHITECTURE currently says “leave it as-is,” but the M1 core (Spec M1-03) raises UnknownTokenError (strict, all-or-nothing). These conflict. Decide: service maps the error to a 4xx (honor strict core), or the core/ARCHITECTURE change to lenient. Recommended: keep the strict core, map to 409/422, and update ARCHITECTURE.

Components

Python sidecar

  • Builds one RedactionAnalyzer at startup; /healthz flips ready after warmup.
  • redact: calls prompt_redact_core.redact(text, token_map, analyzer=…); maps TokenShapedInputError/RedactError to 400.
  • unredact: calls prompt_redact_core.unredact; maps UnknownTokenError per decision #6.

Front-end (Go/TS)

  • Public API, request size cap → 413 (T9), JSON (de)serialization, IPC client to the sidecar, error passthrough/mapping, structured logging without bodies.

API surface

Exactly as in ARCHITECTURE — API surface. Error mapping: malformed body or token-shaped input → 400; body over the cap → 413; unknown-token on unredact → per decision #6; engine failure → 500.

Threats addressed

  • T2/T3 — no request/response bodies in logs; crash handlers scrub.
  • T9 — configurable request-size cap at the front-end.
  • T11 — IPC bound to loopback; mTLS only if ever split across hosts.
  • T4 — documented boundary: the service can't force callers to use it.

Exit criteria (from PLAN)

  • curl round-trips text through /redact and /unredact with stable tokens across calls.
  • Structured logs contain no raw identifiers.
  • Integration test: multi-turn conversation asserts cross-turn token stability + round-trip fidelity.

Proposed spec breakdown (once decisions land)

  1. M2-01 — Python sidecar app + eager analyzer + /healthz.
  2. M2-02 — /redact + /unredact handlers + error mapping + the wire contract.
  3. M2-03 — request size cap + logging discipline.
  4. M2-04 — multi-turn integration test (+ compose if pulled forward).
  5. M2-05 — Go/TS front-end + IPC client (if not shipped as the MVP interim).
