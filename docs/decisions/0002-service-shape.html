ADR 0002 — Service shape: redaction microservice (not LLM proxy)

  • Status: Accepted
  • Date opened: 2026-05-30
  • Owners: rick-johnn
  • Supersedes: The M0 decision recorded as “OpenAI-compatible chat completions endpoint” in PLAN.html's decision log.
  • Superseded by: —

Context

The M0 design picked an OpenAI-compatible chat completions endpoint as the public surface. In that model the service is a transparent proxy: chat clients change their base URL, the service intercepts each request, redacts the prompt, forwards to an upstream LLM, and rehydrates the response on the way back. Every request to the service maps 1:1 to an LLM call.

On 2026-05-30 the user clarified the actual need: not every request makes the LLM call. The redaction step should be invocable on its own, before the chat application decides what to do next (call an LLM, log the text, throw it away, show it back to the user for editing, etc.). The chat application is the orchestrator. Redaction is a service it calls — not a hop traffic transits through.

This is the third option originally surfaced during M0 (“Plain /redact + /unredact endpoints — library-style”), which had been ruled against in favor of the OpenAI-compatible surface. That earlier decision is being reversed.

Decision

The service is a redaction-only microservice. It exposes:

  • POST /redact — accepts text and an optional existing token map; returns redacted text and an updated token map.
  • POST /unredact — accepts text and a token map; returns text with tokens substituted back to originals.
  • GET /healthz — liveness.

It does not:

  • Know about LLM backends. There is no “Upstream client” component.
  • Handle LLM authentication, headers, or schemas.
  • Implement any OpenAI-compatible surface in v1.
  • Maintain server-side session state. Token maps are caller-owned and round-tripped on each call.

The chat application is responsible for: calling /redact before any downstream use of the text; storing the returned token map for the lifetime of the conversation; passing it back on subsequent /redact calls (for cross-turn consistency); and calling /unredact on LLM responses if it wants the user to see real identifiers.

Why the flip

  • Use cases don't map 1:1 to LLM calls. Many redactions happen for non-LLM reasons: showing the user what would be sent, logging to an internal store, indexing for retrieval, archiving. A transparent proxy forces every redaction through an LLM round-trip, which is the wrong shape.
  • Composability. A redaction microservice is a building block. The chat app, a batch job, an audit pipeline, and an eval harness can all call it for their own purposes without each of them having to pretend to be an LLM caller.
  • Smaller threat surface. The service never holds LLM auth tokens, never sees LLM responses, never logs LLM telemetry. Three rows come out of the threat model.
  • Cleaner trust model. The chat application is the trusted orchestrator. The service is a stateless function. There is no implicit guarantee that traffic to an LLM has been redacted — that's the chat app's contract to honor.

Consequences

What's removed

  • The “Upstream client” component in ARCHITECTURE.html.
  • OpenAI-compat schema handling in the service.
  • Streaming SSE handling (M3 in the original plan) — we don't see LLM responses, so there's nothing to stream-rehydrate. /redact input is a string; if very-large inputs become an issue, that's a separate optimization, not v1 scope.
  • Multipart message content concerns (open question #3 in the old ARCHITECTURE) — /redact takes plain text.
  • Threat model rows that depended on the service touching upstream LLMs (T4 upstream telemetry; T7 model-emitted PII; T9 upstream auth tokens).

What stays

  • The redactor itself (Presidio analyzer + token assignment).
  • The token format ([TYPE_N]) and rationale.
  • Per-call (per-request) token-map scoping. Now caller-owned rather than service-held.
  • Redaction quality targets (recall, precision, leakage rate, latency).
  • Hybrid regex + NER performance path.
  • The implementation-language ADR (0001) — Presidio is still Python; the topology choice is unchanged in shape.
  • Deployment target sketch.

What gets stronger

  • The “tampered/forgetful client” threat. Under the old proxy model the service was an unavoidable hop; clients couldn't bypass it without changing their base URL. Under this model the chat app must actively choose to call /redact. A buggy or malicious chat app that forgets the call leaks PII directly to the LLM. The service cannot prevent this — it can only be present when called. Mitigation lives outside the service: code review of chat-app integration paths, optional separate enforcement at the LLM ingress, integration tests that verify the chat app never sends raw input to the LLM.
  • Token-map handling on the caller side. The caller is now responsible for storing and protecting the token map for the lifetime of a conversation. The map contains the raw identifiers (it's literally {token: original}). Treat it as sensitive data on the caller side.

What gets sharper (latency)

Under the old design, ≤ 50 ms p95 for a 500-token message was the added cost on top of an LLM call that already took hundreds of ms. Under this design it's the whole user-perceived round-trip when the chat app calls /redact from a hot path (e.g. on submit, or on debounced typing). Same number is probably still the right target, but it now has less headroom — see ARCHITECTURE — Redaction quality targets.

Alternatives considered (and not picked)

  • Keep OpenAI-compat as a secondary surface alongside /redact. Rejected: doubles the API surface and the threat model for callers who, by the user's own description, don't want it. If a future caller wants transparent proxying, they can build it as a thin wrapper around /redact.
  • Server-held sessions for token-map continuity across turns. Rejected: adds state, eviction policy, a new failure mode, and a new surface to threat-model. Caller-owned maps shift the burden one hop but keep the service stateless and trivially horizontally scalable.
  • Make /unredact the caller's responsibility (return the map from /redact; let callers substitute themselves). Considered but rejected: substitution is non-trivial (token collision, escape rules, partial matches across boundaries). Centralizing it in /unredact keeps the rules in one place.
